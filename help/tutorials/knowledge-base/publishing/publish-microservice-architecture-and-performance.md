# 클라우드 게시 마이크로서비스 아키텍처 및 성능 분석

이 문서에서는 새 클라우드 게시 마이크로 서비스의 아키텍처 및 성능 번호에 대한 통찰력을 공유합니다.

>[!NOTE]
>
> 현재 AEM 가이드의 마이크로서비스 기반 게시는 기본 PDF 게시 또는 DITA-OT를 통한 PDF 출력만 지원합니다. AEM 가이드는 향후 릴리스에서 더 많은 출력 유형에 대한 마이크로서비스 기반 게시 지원을 추가할 예정입니다.

## 클라우드의 기존 게시 워크플로우 문제

DITA Publishing은 주로 사용 가능한 시스템 메모리와 CPU에 따라 리소스를 많이 사용하는 프로세스입니다. 게시자가 많은 주제가 있는 큰 맵을 게시하거나 여러 개의 병렬 게시 요청이 트리거되는 경우 이러한 리소스가 더 필요합니다.

새 서비스를 사용하지 않는 경우 모든 게시는 AEM 클라우드 서버도 실행 중인 동일한 Kubernetes(k8) pod에서 발생합니다. 일반적인 k8 pod에는 사용할 수 있는 메모리 및 CPU의 양에 제한이 있습니다. AEM Guides 사용자가 대규모 또는 병렬 워크로드를 게시하는 경우 이 제한이 빠르게 위반될 수 있습니다. K8은 AEM 클라우드 인스턴스 자체에 심각한 영향을 줄 수 있는 구성된 제한보다 더 많은 리소스를 사용하려고 하는 pod를 다시 시작합니다.

이러한 리소스 제약이 클라우드에서 여러 동시 및 대용량 게시 작업 로드를 실행할 수 있는 전용 서비스를 제안하게 된 주된 동기입니다.

## 새로운 아키텍처 소개

이 서비스는 App Builder, IO Events, IMS와 같은 Adobe의 최신 클라우드 솔루션을 사용하여 서버를 사용하지 않는 오퍼링을 만듭니다. 이러한 서비스는 Kubernetes와 Docker와 같이 널리 받아들여지는 업계 표준에 기반합니다.

새 게시 마이크로 서비스에 대한 각 요청은 한 번에 하나의 게시 요청만 실행하는 분리된 Docker 컨테이너에서 실행됩니다. 새 게시 요청이 수신될 경우 여러 개의 새 컨테이너가 자동으로 만들어집니다. 요청 구성당 이 단일 컨테이너를 통해 마이크로 서비스는 보안 위험 없이 고객에게 최상의 성능을 제공할 수 있습니다. 이러한 컨테이너는 게시가 끝나면 삭제되므로 사용되는 리소스를 모두 사용할 수 있습니다.

이러한 모든 통신은 JWT 기반 인증 및 승인을 사용하여 Adobe IMS에 의해 보호되며 HTTPS를 통해 실행됩니다.

<img src="assets/architecture.png" alt="프로젝트 탭" width="600">

>[!NOTE]
>
> 게시 프로세스는 종속성 목록 생성과 같이, AEM 서버 자체에 대한 요청의 일부 콘텐츠 종속 부분을 실행합니다. 그러나 DITA-OT 실행 또는 기본 엔진 실행과 같은 게시 프로세스의 가장 완전한 부분이 새 서비스에 오프로드되었습니다.


## 성능 분석

이 섹션에서는 마이크로서비스의 성능 번호를 소개합니다. 이전 클라우드 아키텍처가 동시 게시 또는 매우 큰 맵을 게시하는 데 문제가 있었으므로 AEM Guides on-prem 서비스와 마이크로 서비스의 성능을 비교합니다.

on-prem에 큰 맵을 게시하는 경우 Java heap 매개 변수를 조정해야 하거나, 그렇지 않으면 메모리 부족 오류가 발생할 수 있습니다. 클라우드에서 마이크로 서비스는 이미 프로파일링되었으며 최적의 Java 힙과 기타 구성이 즉시 제공됩니다.

### 클라우드와 온프레미에서 한 번의 게시 실행

* 클라우드

   새 서비스를 사용하여 클라우드에서 단일 게시를 실행하는 경우 단일 온-프레미스 게시와 비교하면 게시 시간이 약간 더 걸릴 수 있습니다. 이러한 약간의 상승된 시간은 새로운 클라우드 아키텍처의 분산 특성 때문입니다.

   <img src="assets/cloud_single_publish.png" alt="프로젝트 탭" width="600">

* On-prem

   AEM이 실행되는 동일한 pod/machine에서 전체 게시가 수행되므로 단일 게시 결과는 이전 클라우드 아키텍처나 온프레미스에서 더 좋습니다.

   <img src="assets/onprem_single_publish.png" alt="프로젝트 탭" width="600">

### 클라우드와 온프레미에서 여러 게시 실행

* 클라우드

   이 시나리오에서는 새로운 게시 마이크로 서비스가 표시됩니다. 아래 이미지에서 볼 수 있듯이 여러 동시 게시 작업이 증가하면 클라우드에서 게시 시간을 크게 늘리지 않고도 게시할 수 있습니다.

   <img src="assets/cloud_bulk_publish.png" alt="프로젝트 탭" width="600">

* On-prem

   온-프레미스 서버에서 동시 게시를 실행하면 성능이 심각하게 저하됩니다. 게시자가 더 많은 맵을 동시에 게시하는 경우 이 성능 저하 가 더 심각합니다.

   <img src="assets/onprem_bulk_publish.png" alt="프로젝트 탭" width="600">

## 추가적인 이점

마이크로 서비스로 전송할 올바른 게시 컨텐츠를 가져오려면 각 게시 요청의 일부 부분이 AEM 인스턴스에서 실행되어야 합니다. 새 클라우드 아키텍처는 이전 아키텍처에서와 마찬가지로 AEM 워크플로우 대신 AEM 작업을 사용합니다. 이 변경 사항을 통해 AEM 안내서 관리자는 다른 AEM 작업 또는 워크플로우 구성에 영향을 주지 않고 개별적으로 클라우드 게시 큐 설정을 구성할 수 있습니다.

새 게시 마이크로 서비스를 구성하는 방법에 대한 자세한 내용은 다음 위치에서 확인할 수 있습니다. [마이크로서비스 구성](configure-microservices.md)
